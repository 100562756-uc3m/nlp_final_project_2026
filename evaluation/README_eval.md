# RAG System Evaluation — DailyMed NLP Project (UC3M 2026)

## Overview

This module provides a dual-layered evaluation of the DailyMed RAG system. It is designed to isolate and measure the two critical phases of the pipeline:

1. **Retrieval Evaluation:** Measures the ability of the FAISS vector index to find correct clinical documents across a grid of hyperparameters (K and Threshold).

2. **Generation Evaluation:** Uses an "LLM-as-a-Judge" approach to audit the quality, safety, and medical accuracy of the final answers generated by Llama 3.1:8b.

---

## 1. Retrieval Evaluation (Grid Search)

The retrieval evaluation is retrieval-only — it does not call the LLM for answers. This isolates retrieval errors from generation errors: even a perfect LLM cannot produce a good answer if the retrieval step fails to find the correct clinical context.

### Evaluation Criteria

| Criterion | What it measures | How it is computed |
|---|---|---|
| **RAG System Quality** | Whether the system retrieves relevant documents and ranks them well | Hit@K, Recall@K, Precision@K, MRR |
| **Document Coverage** | Fraction of queries answered using database information | Hit@K across all answerable questions |
| **Response Time** | Speed at which retrieval completes | Average and P95 latency in milliseconds |


### Metrics

All metrics are computed by comparing the **retrieved documents** (drug name + section title) against the **expected support** defined in the evaluation set.

Matching uses fuzzy string comparison (`difflib.SequenceMatcher`, threshold 0.75) plus a section title mapping dictionary that normalises FDA section names (e.g. `"warnings and precautions"`) to internal group labels (e.g. `"safety_risk"`). This makes the evaluation robust to minor naming differences between the eval set and the index.

#### Hit@K
Binary: did at least one retrieved document match any expected document?

```
Hit@K = 1 if |relevant ∩ top-K| ≥ 1, else 0
```

Aggregated as the fraction of questions where at least one relevant document was found.

#### Recall@K
Fraction of relevant documents found in the top-K results. Answers: *did we retrieve everything we needed?*

```
Recall@K = |relevant ∩ top-K| / |relevant|
```

#### Precision@K
Fraction of the top-K retrieved documents that are actually relevant. Answers: *how much noise are we sending to the LLM?*

```
Precision@K = |relevant ∩ top-K| / K
```

#### MRR (Mean Reciprocal Rank)
Rewards systems that rank the best result first. Answers: *is the best result at the top?*

```
MRR = (1/|Q|) × Σ (1 / rank_i)
```

Where `rank_i` is the position (1-indexed) of the first relevant document for query `i`.

### Latency Metrics

| Metric | Description |
|---|---|
| `lat_avg_ms` | Average retrieval time across all queries |
| `lat_p95_ms` | 95th percentile latency — the worst-case speed for 95% of queries |


## 2. Generation Evaluation (LLM-as-a-Judge)

The generation evaluation audits the end-to-end pipeline. It uses a high-capability LLM (the "Judge") to act as a Medical Quality Auditor to score the chatbot's synthesized responses.

### The "LLM-as-a-Judge" Workflow

1. **Pipeline Run:** Every question in the eval set is processed through the full `get_bot_response` function using the optimal parameters (K=8, T=0.40).

2. **Auditor Prompt:** The auditor receives the Question, the Retrieved Context, and the Generated Answer.

3. **Scoring:** The auditor provides a score from 1 to 5 based on strict clinical rubrics.

### Generation Metrics

| Metric | What it measures | Clinical Significance |
|---|---|---|
| **Groundedness** | Is the answer fully supported by the context? (1=Hallucination, 5=Perfectly Grounded) | Prevents the AI from making up medical facts not in the FDA label. |
| **Relevance** | Does it accurately answer the user's specific question? (1=Irrelevant, 5=Perfectly Relevant) | Ensures the user receives the specific information requested. |
| **Refusal Accuracy** | Percentage of correct refusal decisions. | Validates safety guardrails (e.g., refusing personal medical advice). |


## Evaluation Grid (Retrieval)

The script runs every combination of:

| Parameter | Values tested |
|---|---|
| **K** (top-k retrieved) | 3, 5, 8, 10 |
| **Threshold** (min cosine similarity) | 0.20, 0.30, 0.40, 0.60, 0.80 |

This produces **20 combinations** in total. For each combination the script reports metrics for:
- All questions
- English-only questions
- Non-English questions (translated to English before retrieval, matching the production behaviour of `system.py`)

---

## Similarity Score

Retrieval uses **cosine similarity** derived from the L2 distance returned by FAISS. With normalised embeddings the relationship is exact:

```
cosine_similarity = 1.0 - (L2_distance / 2.0)
```

A threshold of 0.25 means only documents with at least 25% cosine similarity to the query are considered. Documents below the threshold are discarded before any metric is computed.

## Evaluation Set Format

The eval set is a JSONL file where each line is a JSON object:

```json
{
  "id": "q001",
  "question": "What are the contraindications of Aspirin?",
  "language": "en",
  "expected_refusal": false,
  "expected_support": [
    {"drug_name": "Aspirin", "section_title": "contraindications"},
    {"drug_name": "Aspirin", "section_title": "warnings and precautions"}
  ]
}
```

| Field | Description |
|---|---|
| `id` | Unique identifier |
| `question` | The query (any language) |
| `language` | ISO language code (`"en"`, `"es"`, `"fr"`, etc.) |
| `expected_refusal` | `true` if the correct answer is "I don't have information" |
| `expected_support` | List of documents the retriever should find (only for answerable questions) |

Questions with `expected_refusal: true` are excluded from retrieval metrics — they are used only for generation-level refusal accuracy, which is evaluated separately.

## How to Run the Evaluation

### 1. Run the Evaluation Grid

```bash
python -m scripts.evaluate_rag \
  --eval-set evaluation/dailymed_eval_v2.jsonl \
  --index data/vector_db/smart_index/index.faiss \
  --meta data/vector_db/smart_index_inspect.jsonl \
  --output-dir evaluation/grid_results
```

### 2. Run Generation Quality Audit

```bash
python -m scripts.evaluate_generation
```

### 3. Launch the Dashboard

After running the evaluation, launch the Streamlit dashboard to view tables and line charts interactively:

```bash
streamlit run evaluation/dashboard_unified.py
```


The dashboard shows:
- **A filterable comparison table**: Users can filter the global grid by Top-K and Similarity Threshold to see specific performance metrics.
- **Line charts for Retrieval**: Visualizes trends for Recall@K, Precision@K, MRR, and Average Latency (ms) across different configurations.
- **Generation Quality Distributions**: Bar charts showing the frequency of Groundedness and Relevance scores (1-5) for answerable queries.
- **Refusal Accuracy Donut**: A safety chart visualizing the percentage of correct versus incorrect refusal decisions.
- **Language Performance**: A side-by-side comparison of retrieval and generation quality between English and foreign language queries.
- **Download buttons**: Option to export both the retrieval grid and the detailed generation audit as CSV files.


## Interpreting Results

### Retrieval Performance
- **High Recall@K, low Precision@K** → The system successfully finds the required clinical documents but also retrieves "noise" (irrelevant chunks). This forces the LLM to process more text, increasing token costs and potential confusion. Consider raising the threshold.
- **High MRR** → The most relevant clinical context is being ranked at the top of the results. This is ideal for LLMs, as they often exhibit "lost in the middle" bias and perform better when the most pertinent information appears first.
- **Hit@K drops sharply as threshold increases** → If success rates fall significantly after a specific threshold (e.g., 0.6), it indicates that the semantic similarity between user queries and medical labels typically sits in the 0.4–0.5 range.
- **Foreign language gap** (`hit@k_en` vs `hit@k_others`) → If metrics for non-English queries are lower, the LLM-based translation step in `system.py` may be losing clinical nuance before the search occurs.

### Generation & Safety Quality
- **Average Groundedness (< 4.0)** → A low groundedness score suggests the model is "hallucinating" facts not present in the DailyMed sources or relying too heavily on its internal training data. This requires stricter system prompting.
- **Average Relevance (< 4.0)** → The model may be providing generic answers or failing to address the specific nuance of the user's medical query, even if the correct documents were retrieved.
- **Refusal Accuracy** → This is the primary safety metric. 
    - **False Positives**: The bot refuses a valid question it has data for (the threshold is likely too high).
    - **False Negatives**: The bot answers a dangerous or out-of-scope question it should have refused (the threshold is too low or safety prompts are weak).
- **Latency (Generation)** → Total pipeline latency (averaging ~5.8s) includes translation, retrieval, and LLM generation. If this spikes, check the UC3M API response times or the number of retrieved tokens (Top-K).
